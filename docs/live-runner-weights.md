# Live runner weights: neural value net vs tabular Q (MDP/POMDP), and how to pull them into `:5055`

This explains the two *separate* learned components the soccer engine produces, and how
to load them into a live inference runner such as `localhost:5055` (`dd-soccer-rs`, or the
engine's own `main_soccer_live`). Grounded in the running continuous learner
(`main_soccer_learning_run`, experiment slug `soccer-self-play-k8s-overnight`).

## The learned components are independent

The engine does **not** produce one monolithic "policy." Several things are learned in
parallel and stored together per generation:

| Component | What it is | How it's learned | Where stored |
|---|---|---|---|
| **Neural value net** | A small feed-forward net: state features (`inputDim = SOCCER_NEURAL_FEATURE_DIM` in the runner build) â†’ **one scalar value** (`outputDim = 1`). Default `hidden_units = 24`, with the live `parameterCount` reported in metrics. It *scores states* and generalizes â€” it is **the primary decision driver**, but it is **not an action policy by itself**. | Gradient descent (mini-batches + replay, a training loss) during self-play. | `des_soccer_learning_policy_versions.metrics -> neuralNetwork` (`inputDim`, `outputDim`, `layers`, `parameterCount`, `l2Norm`, `trainingSteps`). |
| **Tabular Q policy (MDP / POMDP)** | A table keyed by a discretized MDP-state / POMDP-observation hash â†’ action â†’ **Q-value + visit count**. This is the classic MDP/POMDP action-value table. | Tabular RL updates (visit-count value updates from game rewards). **Not generated by the neural net** â€” a separate learner. | `des_soccer_learning_policy_entries` (millions of rows per generation). |
| **Tactical weights (MARL / MAPPO)** | Team-level shaping deltas (attack/defense width, spacing, press, flank-lane, LP guidance). | MARL/MAPPO team-reward-shared updates. | version `.config` / `.metrics -> tacticalLearning`. |
| **Decision gates + MPC/QP** | Hand-coded behaviour rules (`DD_SOCCER_ENABLE_*`) and the MPC/QP execution layer that turns a chosen intent into kinematics. | Not learned (gates) / analytic (MPC). | env / code. |

> **Key nuance â€” the tabular Q is a *weak, sparse prior*, not the brain.**
> The MDP/POMDP state space is very high-dimensional, so `visits/entry â‰ˆ 1.0` â€” most states
> are seen ~once. The tabular table is therefore mostly *memorised single states* that rarely
> match a live game state. The **neural value net is the component that actually generalises.**
> See `soccer.rs` (`SOCCER_NEURAL_FEATURE_DIM`, and the comment: *"The NEURAL net is the primary
> decision driver now; the tabular value is only a weak prior"*).

So "where does the learning live?" â†’ mostly the **neural value net + tactical weights**. The
tabular Q is currently near-noise at this state-dim; loading it does **not** make behaviour
smarter. On mature generations it can be millions of rows, so loading it is slow and heavy.

## How weights flow to Postgres

`main_soccer_learning_run` plays self-play games and, per generation, writes a **policy
version** to Postgres (experiment keyed by `SOCCER_EXPERIMENT_SLUG`):
`des_soccer_learning_policy_versions` (neural snapshot + tactical in JSON) +
`des_soccer_learning_policy_entries` (the tabular rows). A version is promoted to
`status = 'active'` only if it clears the promotion gate
(`SOCCER_BATCH_POLICY_PROMOTION_MAX_MEAN_*`). Schema source of truth:
`~/codes/ores/k8s-cluster/remote/libs/pg-defs`.

## Pulling weights into a live runner (`:5055`)

The runner **must be built with the `postgres-persistence` feature** â€” otherwise every PG
code path is `#[cfg]`-compiled out and it silently serves a local-file fallback (a default,
unlearned policy). For `dd-soccer-rs`: `soccer_engine = { path = "...", features = ["postgres-persistence"] }`.

Then set env:

```bash
SOCCER_DATABASE_URL=postgres://â€¦                 # RDS url (Secrets Manager: dd/remote-dev/rest-api-secrets -> RDS_DATABASE_URL)
SOCCER_PG_TLS_INSECURE=1
SOCCER_EXPERIMENT_SLUG=soccer-self-play-k8s-overnight   # MUST match the learner's slug
SOCCER_LIVE_POLICY_INCLUDE_UNPROMOTED=1           # show newest BEST CANDIDATE (generation desc, fitness desc),
                                                  #   not only the last promotion-gated `active`
SOCCER_LIVE_POLICY_NEURAL_TACTICAL_ONLY=true      # RECOMMENDED: load neural value head + tactical only (fast).
                                                  #   This is the primary driver; the sparse tabular is skipped.
# SOCCER_LIVE_POLICY_MIN_VISITS=1                 # only used when NEURAL_TACTICAL_ONLY is false: also load tabular
                                                  #   entries with visits>=N. At current state-dim this is ~2M rows,
                                                  #   slow + memory-heavy, and does NOT improve behaviour. Avoid.
# SOCCER_LIVE_POLICY_PG_REFRESH_SECONDS=60        # background thread hot-swaps a newer/better gen every 60s (default)
```

Startup is **bind-first**: the server binds immediately (serving the local-file fallback),
then a background thread performs the Postgres load within seconds and logs
`# soccer-live: refreshed to learned policy gen N from postgres`. Every new per-game session
clones the warm in-memory policy (`reused warm in-memory learned policy`).

Local helper: **`~/bin/soccer-live-5055.sh`**. UI: **http://localhost:5055/soccer**
(the bare root `/` is not routed on `dd-soccer-rs` â€” the UI lives under the `/soccer` prefix;
`main_soccer_live` serves at `/`).

## Verifying the weights actually loaded (introspection)

- `GET /soccer/inspect?game=<id>&weights=1` â†’
  - `â€¦neuralNetwork.parameterCount` **> 0** and `inputDim` **==** the runner's
    compiled `SOCCER_NEURAL_FEATURE_DIM` â‡’ neural head loaded. If `inputDim` differs,
    the warm-start drops the snapshot (`dropping incompatible neural
    snapshot â€¦ does not match expected`) and the net cold-starts â€” keep runner + learner on
    the same engine version.
- `GET /soccer/api/state?game=<id>` â†’ the `learning` / `policyProbability` block:
  - `homePolicyEntries` / `awayPolicyEntries` â€” tabular row count (**0 in `NEURAL_TACTICAL_ONLY`
    mode â€” expected**).
  - `policyProbability.homeStates` / `homeActions` â€” size of the learned action policy in play.
  - `tacticalSummary.*` â€” MARL/MAPPO shaping (accumulates during play).

## Caveats (why a correctly-loaded runner can still look unintelligent)

- **Model maturity dominates.** `hidden_units = 24` is a tiny value net, and the tabular is
  sparse. An under-trained generation looks basic no matter how correctly it's loaded. Real
  quality comes from many more training generations (and, longer-term, a bigger net).
- **Don't load the tabular to "fix" behaviour** â€” it's a weak/sparse prior at this state-dim.
- **Feature-dim drift** (a code change to `SOCCER_NEURAL_FEATURE_DIM`) cold-resets the neural
  head on the next resume. Only *tail-appended* feature blocks are pad-migratable
  (`SOCCER_NEURAL_LEGACY_FEATURE_DIMS`); a mid-vector reshape forces a cold start.
